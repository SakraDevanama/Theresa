# Theresa Mod 微尘系统联机同步问题总结

## 当前问题：MaoCrest `IsFull` 联机状态分歧（第一回合即分歧）

### 现象

联机模式下第一回合开始即出现状态分歧：
- **Host 看到的 Client**：Hand=5 张牌, Block=0, Mantra=0
- **Client 看到的自己**：Hand=3 张牌, Block=5, Mantra=2
- **分歧时机**："After player turn start"

### 根因分析（2026-05-06 更新）

#### 问题本质：`DustManager.IsFull` 使用 `GetLocalTheresaPlayer()`，在 Host 端检查的是错误的玩家

`MaoCrest.AfterPlayerTurnStart` 使用 `DustManager.IsFull`（无参属性）判断微尘是否已满：
```csharp
if (DustManager.IsFull && mantraConverted < maxMantraConversions)
```

`IsFull` 内部实现：
```csharp
public static bool IsFull => Count >= MaxDust;
public static int Count => GetDustCardsFromExhaust(GetLocalTheresaPlayer()).Count;
public static int MaxDust => GetMaxDust(GetLocalTheresaPlayer());
```

**问题**：`GetLocalTheresaPlayer()` 返回**本地** Theresa 玩家。在联机 Host 端处理 Client（Theresa）玩家的回合时：
- Host 本地玩家是 Ironclad，不是 Theresa
- `GetLocalTheresaPlayer()` 返回 null 或 Host 自己（如果 Host 也是 Theresa）
- `Count = 0`, `MaxDust = 3`
- `IsFull = false` → Host 始终执行抽牌，从不执行 Mantra 转化

**而 Client 端**：
- `GetLocalTheresaPlayer()` 正确返回 Client 自己
- 初始抽牌后 dust=3/3 → `IsFull = true`
- Client 执行 Mantra 转化（获得 2 层 MantraPower → Block=5）

**两端执行路径不同**：
| 端 | MaoCrest 第1次 | 第2次 | 第3次 | 结果 |
|---|---|---|---|---|
| Host | Draw | Draw | Draw | Hand+3, Mantra=0 |
| Client | Mantra | Mantra | Draw | Hand+1, Mantra=2 |

这导致手牌分布和 Power 状态严重分歧。

### 修复

1. **`DustManager`**：添加 `IsFullForPlayer(Player player)` 和 `GetCount(Player player)` 方法
2. **`MaoCrest`**：使用 `DustManager.IsFullForPlayer(player)` 替代 `DustManager.IsFull`
3. **`MaxDustVar`**：使用 `_owner` 获取 relic 的 Owner 玩家，替代 `DustManager.MaxDust`

**代码修改**（`DustManager.cs`）：
```csharp
public static bool IsFullForPlayer(Player player)
{
    if (player == null) return false;
    return GetCardsForPlayer(player).Count >= GetMaxDust(player);
}

public static int GetCount(Player player)
{
    if (player == null) return 0;
    return GetDustCardsFromExhaust(player).Count;
}
```

**代码修改**（`Relics/MaoCrest.cs`）：
```csharp
// 修复前：
if (DustManager.IsFull && mantraConverted < maxMantraConversions)

// 修复后：
if (DustManager.IsFullForPlayer(player) && mantraConverted < maxMantraConversions)
```

### 教训

**所有 `DustManager` 的无参属性（`IsFull`, `Count`, `MaxDust`, `Cards`）都只适用于本地玩家，联机代码中必须使用带 `Player` 参数的版本。**

## 当前问题：WishSilkEnchantment 使用 `DustManager.Cards` 导致联机分歧

### 现象

`WishSilkEnchantment`（愿景丝线）附魔的牌被抽到时，会检查微尘中是否已有此牌的副本。联机模式下 Host/Client 检查结果不同，导致一端转化、一端不转化 → 状态分歧。

### 根因

`WishSilkEnchantment.ConvertToDustAsync()` 和 `CanEnchant()` 使用 `DustManager.Cards`（无参属性）：
```csharp
var hasCopyInDust = DustManager.Cards.Any(c => c.Owner == Card.Owner && ...);
```

`DustManager.Cards` 内部使用 `GetLocalTheresaPlayer()`，在 Host 端返回 Host 自己（Ironclad），不是 Client（Theresa）。Host 端 Ironclad 没有微尘 → `hasCopyInDust = false` → 总是转化。Client 端正确检查 → 可能已有副本不转化。

### 修复

使用 `DustManager.GetCardsForPlayer(Card.Owner)` 替代 `DustManager.Cards`：
```csharp
var dustCards = Card.Owner != null ? DustManager.GetCardsForPlayer(Card.Owner) : new List<CardModel>();
var hasCopyInDust = dustCards.Any(c => c.Owner == Card.Owner && ...);
```

## 当前问题：ThousandsWishPower fire-and-forget 导致联机时序分歧

### 现象

`ThousandsWishPower` 达到抽牌阈值后触发额外萦绕，但使用 `_ = TriggerExtraLingerDelayed(...)` fire-and-forget。额外萦绕在 `AfterCardDrawn` 返回后异步执行，可能在 checksum 检测时一端已完成、另一端未完成 → 状态分歧。

### 根因

```csharp
_ = TriggerExtraLingerDelayed(lingerCount);  // fire-and-forget

private async Task TriggerExtraLingerDelayed(int lingerCount)
{
    await Cmd.Wait(0.8f);  // 延迟 0.8 秒
    // ... 执行萦绕
}
```

额外萦绕在抽牌动画完成后 0.8 秒执行。如果 Host/Client 的动画速度不同，或 `DustItFromAction` 中的 `AutoPlay` 克隆牌移除时机不同，checksum 检测时两端状态不一致。

### 修复

1. **移除 fire-and-forget**：使用 `await TriggerExtraLinger(...)` 替代 `_ = TriggerExtraLingerDelayed(...)`
2. **移除 `Cmd.Wait(0.8f)`**：立即执行，减少时序不确定性
3. **`AtTurnStartPostDraw` 增加等待**：`await Cmd.Wait(0.3f)` 确保正常萦绕的克隆牌动画完成

```csharp
// ThousandsWishPower.cs
await TriggerExtraLinger(lingerCount);  // 同步执行

// DustManager.cs - AtTurnStartPostDraw
await DustIt(player, false, false, choiceContext);
await Cmd.Wait(0.3f);  // 等待克隆牌动画完成
```

## 历史问题：UnknownRelic ActionCount 分歧 (136 vs 137)

### 现象

联机模式下出现状态分歧，具体表现为：
- **UnknownRelic ActionCount 不一致**：Host=136, Client=137（差1）
- **分歧时机**："After player turn phase one end"（玩家回合阶段一结束时）
- **PlayerChoiceContext 栈错误**（历史问题，4月22日已修复）：
  ```
  Tried to pop CARD.THERESA-SPOT from stack but POWER.THERESA-KAZDEL_IN_FLAMES_EFFECT was on top instead!
  ```

### 根因分析（2026-04-22 更新）

#### 1. 问题本质：KnownRelicActionCounterPatch 在联机模式下计数不一致

`KnownRelicActionCounterPatch` 拦截了三个入口来计数：
1. `GameAction.Execute` — 走 Action 队列的操作 ✅ **联机安全**
2. `CombatHistory.CardDrawn` — 每张牌被抽到时 ⚠️ **联机不安全**
3. `CardCmd.AutoPlay` — 自动打牌（固有微尘、FINALE 等）⚠️ **联机不安全**

**问题**：入口 2 和 3 在联机模式下不走 `GameAction` 队列，导致两端执行次数可能不一致：
- **入口 2 (`CardDrawn`)**：`DustDrawPatch` 直接调用 `CombatHistory.CardDrawn()`，不走 Action 队列
- **入口 3 (`AutoPlay`)**：`DustItFromAction` 直接调用 `CardCmd.AutoPlay()`，不走 Action 队列

在联机模式下，只有 `GameAction` 是跨端同步的。非 `GameAction` 的操作在 Host 和 Client 两端独立执行，如果初始状态有微小差异（如 RNG 状态、卡牌引用等），就可能导致执行次数不同，进而导致 ActionCount 分歧。

#### 2. 为什么差 1？

从日志分析，分歧发生在 Host (76561198972953168) 和 Client (76561198972953169) 之间：
- Host 本地状态：玩家 76561198972953168 的 ActionCount = 136
- Client 看到的 Host 状态：玩家 76561198972953168 的 ActionCount = 137

这说明 **Client 比 Host 多计数了一次**。最可能的原因是：
- 某个 `AutoPlay` 或 `CardDrawn` 操作在 Client 端被计数，但在 Host 端没有被计数（或 vice versa）
- 例如：克隆卡牌的 `Owner` 属性在两端不一致，导致 `TryIncrementForPlayer(card.Owner)` 在一端执行，另一端跳过

#### 3. SPOT + KazdelInFlames + ZaakathHate 的交互（历史问题）

SPOT 打出时的触发链：
```
SPOT.OnPlay()
  ├─ DamageCmd.Attack() → 造成伤害
  │   └─ 触发 Hook.AfterCardPlayed
  │       └─ KazdelInFlamesEffect.AfterCardPlayed()
  │           └─ PowerCmd.Apply<ZaakathHatePower>()
  │               └─ ZaakathHatePower.AfterPowerAmountChanged()
  │                   └─ CreatureCmd.Damage() [BlockingPlayerChoiceContext]
  │                       → 对所有敌人造成恨意等量伤害
  └─ ChooseDustToHand() → 从微尘中选牌加入手牌
```

**4月22日已修复**：将 `ThrowingPlayerChoiceContext` 替换为 `BlockingPlayerChoiceContext`，解决了 PlayerChoiceContext 栈管理错误。

### 修复方向

#### 方案 C（已实施）：修复 KnownRelicActionCounterPatch 的计数逻辑

**核心思路**：联机模式下只保留 `GameAction.Execute` 入口，跳过 `CardDrawn` 和 `AutoPlay` 入口。

**原因**：
- `GameAction` 在联机模式下是跨端同步的，使用 `OwnerId` 可以正确识别操作发起者
- `CardDrawn` 和 `AutoPlay` 不走 Action 队列，两端执行次数可能不一致
- 如果某个操作需要计数，它应该通过 `GameAction` 执行

**代码修改**（`Patches/KnownRelicActionCounterPatch.cs`）：
```csharp
// 联机模式下跳过非 GameAction 入口
[HarmonyPatch(typeof(CombatHistory), nameof(CombatHistory.CardDrawn))]
public static class CombatHistoryCardDrawnPatch
{
    [HarmonyPostfix]
    public static void Postfix(...)
    {
        if (IsMultiplayer()) return;  // ← 联机模式下跳过
        // ...
    }
}

[HarmonyPatch(typeof(CardCmd), nameof(CardCmd.AutoPlay))]
public static class CardCmdAutoPlayPatch
{
    [HarmonyPostfix]
    public static void Postfix(...)
    {
        if (IsMultiplayer()) return;  // ← 联机模式下跳过
        // ...
    }
}
```

**副作用**：
- 联机模式下 ActionCount 计数可能略低于单机模式（因为遗漏了一些非 GameAction 操作）
- 但这比状态分歧（导致游戏断开）要好得多
- 如果精确计数很重要，可以将抽牌和 AutoPlay 操作包装为 GameAction

## 修复记录

### 2026-04-24: `DUST_IN_HAND` 进入微尘牌堆被萦绕打出导致分歧

**问题**：`DUST_IN_HAND`（执尘）是技能牌，会被 `DustDrawPatch` 抽牌时转化为微尘。萦绕 (`DustIt`) 从微尘牌堆中随机选择卡牌自动打出，如果选中 `DUST_IN_HAND`，其 `OnPlay` 会触发抽牌+手牌转化，导致 Host/Client 状态严重分歧。

**日志证据**：
```
[DustManager] SelectDustCardAndTarget: player=68 card=THERESA-DUST_IN_HAND
[DustItFromAction] player=68 card=THERESA-DUST_IN_HAND
[DustInHand] Auto-selecting from DustIt: THERESA-DUST_IN_HAND
[DustInHand] Enqueue convert action: index=0, card=THERESA-DUST_IN_HAND
DrawCardsAction: Converting to NetAction for player 68, count=2
```

**修复**：
1. `DustInHand.cs`：给 `DUST_IN_HAND` 添加 `DimKeyword.Dim`（黯淡）标签
   - `DustDrawPatch` 抽牌转化时跳过此牌
   - `ShouldBecomeDust` 返回 false
   - 手牌选择转化时也被过滤
2. `DustManager.cs`：`SelectDustCardAndTarget` 和 `DustItWithSelection` 中排除 `THERESA-DUST_IN_HAND`（保险措施）

### 2026-04-23 (3): `DustManager.MaxDust` 联机状态分歧修复

**问题**：`DustManager.MaxDust`（无参属性）内部调用 `GetMaxDust(GetLocalTheresaPlayer())`。在联机模式下，Host 和 Client 的"本地玩家"不同（例如 Host=76561198972953169, Client=76561198972953168），导致 MaxDust 检查产生分歧。这进一步导致 `DustDrawPatch` 在两端的 dust 转化决策不同，引发状态分歧。

**相关文件及修复**：

| 文件 | 修复前 | 修复后 |
|------|--------|--------|
| `Relics/MaoCrest.cs` | `DustManager.MaxDust` | `DustManager.GetMaxDust(relic.Owner)` |
| `Cards/TowardEndfield.cs` | `DustManager.MaxDust` | `DustManager.GetMaxDust(card.Owner)` |
| `Cards/Wish.cs` | `DustManager.MaxDust` | `DustManager.GetMaxDust(Owner)` |
| `Relics/BabelWord.cs` | `DustManager.MaxDust` | `DustManager.GetMaxDust(player)` |

**代码修改**（`Relics/MaoCrest.cs`）：
```csharp
public class MaxDustVar() : DynamicVar("MaxDust", 0m)
{
    protected override decimal GetBaseValueForIConvertible()
    {
        // 优先从 Owner 获取玩家，避免联机下使用错误的本地玩家
        if (_owner is RelicModel relic && relic.Owner != null)
        {
            return DustManager.GetMaxDust(relic.Owner);
        }
        // 角色选择界面时战斗未开始，GetLocalTheresaPlayer 可能为 null
        try { return DustManager.MaxDust; }
        catch { return DustManager.BaseMaxDust; }
    }

    public override string ToString()
    {
        if (_owner is RelicModel relic && relic.Owner != null)
        {
            return DustManager.GetMaxDust(relic.Owner).ToString();
        }
        try { return DustManager.MaxDust.ToString(); }
        catch { return DustManager.BaseMaxDust.ToString(); }
    }
}
```

**注意**：`MaxDustModifierField` 是基于 `ConditionalWeakTable` 的本地字段，**不跨网络同步**。MaxDust 的修改必须通过所有客户端都会执行的同步代码路径（如 `GameAction`、Hook）来触发，确保两端一致。

**待验证**：需要联机测试确认此修复是否解决了 dust 转化决策分歧问题。

### 2026-04-22 (2): KnownRelicActionCounterPatch 联机安全修复

**问题**：`KnownRelicActionCounterPatch` 的 `CombatHistory.CardDrawn` 和 `CardCmd.AutoPlay` 入口在联机模式下不走 `GameAction` 队列，导致两端计数不一致，引发状态分歧。

**修复**（`Patches/KnownRelicActionCounterPatch.cs`）：
- 联机模式下跳过 `CardDrawn` 和 `AutoPlay` 入口的计数
- 只保留 `GameAction.Execute` 入口（联机安全）
- 单机模式下三个入口继续正常工作

**原理**：只有 `GameAction` 在联机模式下是跨端同步的。非 `GameAction` 操作（如直接调用 `CardCmd.AutoPlay`）在两端独立执行，容易因状态差异导致计数不一致。

**待验证**：需要联机测试确认此修复是否解决了 ActionCount 分歧问题。

### 2026-04-22 (1): ZaakathHatePower 使用 BlockingPlayerChoiceContext

**问题**：`ZaakathHatePower.AfterPowerAmountChanged` 使用 `new ThrowingPlayerChoiceContext()` 调用 `CreatureCmd.Damage()`。`ThrowingPlayerChoiceContext` 的 `SignalPlayerChoiceBegun/Ended` 会抛出 `NotImplementedException`，在联机模式下可能导致 PlayerChoiceContext 栈管理混乱，进而引发状态分歧。

**修复**（`Powers/ZaakathHatePower.cs`）：
将 `ThrowingPlayerChoiceContext` 替换为 `BlockingPlayerChoiceContext`：

```csharp
// 修复前：
await CreatureCmd.Damage(
    new ThrowingPlayerChoiceContext(),
    enemy,
    damageAmount,
    ValueProp.Unpowered,
    Owner,
    null
);

// 修复后：
var damageContext = new BlockingPlayerChoiceContext();
await CreatureCmd.Damage(
    damageContext,
    enemy,
    damageAmount,
    ValueProp.Unpowered,
    Owner,
    null
);
```

**原理**：`BlockingPlayerChoiceContext` 的 `SignalPlayerChoiceBegun/Ended` 都是 `Task.CompletedTask` 空操作，不会抛出异常，也不会干扰 Action 队列的暂停/恢复机制。这在联机模式下更安全。

## 已修复的问题历史

### 1. 单机微尘系统不工作
- **原因**：`_dustCandidates` 使用 ReferenceEquals 匹配卡牌，回合间引用变化后匹配失败
- **修复**：替换为 `SpireField<CardModel, bool>` 标记系统

### 2. 联机萦绕触发两次
- **原因**：MaoCrest.AfterPlayerTurnStart 在 Host 和 Client 都执行
- **修复**：移除 LingerDustAction 包装，直接调用 DustItFromAction

### 3. UnknownRelic ActionCount 分歧（第一版）
- **原因**：KnownRelicActionCounterPatch 使用 `LocalContext.GetMe()` 在联机下返回错误玩家
- **修复**：改用 `GameAction.OwnerId` 确定正确的玩家

### 4. MantraPower 数量分歧
- **原因**：AddCardAtIndex 添加的卡牌不在初始 `_dustCandidates` 中
- **修复**：使用 SpireField 标记替代候选列表

### 5. 查看其他玩家微尘牌堆显示自己的牌
- **原因**：UI 使用本地玩家的微尘卡牌而非目标玩家
- **修复**：NDustRing 追踪 `_ownerPlayer`，使用 `GetCardsForPlayer`

### 6. "Card must be in a combat pile" 错误
- **原因**：CardPileCmd.RemoveFromCombat 对已被 AutoPlay 移除的克隆牌调用
- **修复**：包装 try-catch 处理已移除的卡牌

### 7. 第二回合不再拦截卡牌进入微尘
- **原因**：`_dustCandidates` 引用比较失败
- **修复**：使用 SpireField 标记系统

### 8. PlayerChoiceContext 栈管理错误
- **原因**：ZaakathHatePower 使用 ThrowingPlayerChoiceContext，在联机下抛出异常干扰栈管理
- **修复**：替换为 BlockingPlayerChoiceContext

### 9. UnknownRelic ActionCount 分歧（第二版）
- **原因**：KnownRelicActionCounterPatch 的 CardDrawn 和 AutoPlay 入口在联机下不走 GameAction 队列，导致两端计数不一致
- **修复**：联机模式下只保留 GameAction.Execute 入口

### 10. DustManager.MaxDust 联机状态分歧
- **原因**：`DustManager.MaxDust` 使用 `GetLocalTheresaPlayer()`，联机下 Host/Client 本地玩家不同，MaxDust 检查产生分歧
- **修复**：所有使用 `DustManager.MaxDust` 的地方替换为 `DustManager.GetMaxDust(specificPlayer)`

### 11. TenRings 使用私有计数器
- **原因**：`TenRings` 使用私有 `_turnCounter`，不跨网络同步
- **修复**：改用同步的 `CombatState.RoundNumber`

### 12. DustInHand 联机重复发送 Action
- **原因**：`DustInHand.OnPlay` 在两端都执行，重复发送 GameAction
- **修复**：只在 `LocalContext.IsMe(Owner)` 时发送 Action

### 13. MaoCrest `IsFull` 联机状态分歧（第一回合即分歧）
- **原因**：`MaoCrest.AfterPlayerTurnStart` 使用 `DustManager.IsFull`（无参属性），内部调用 `GetLocalTheresaPlayer()`。在联机 Host 端处理 Client 玩家回合时，`GetLocalTheresaPlayer()` 返回 Host 自己（Ironclad）→ `Count=0` → `IsFull=false` → Host 始终抽牌。Client 端正确检查 → dust 满时执行 Mantra。两端执行路径不同导致分歧。
- **修复**：
  - `DustManager` 添加 `IsFullForPlayer(Player player)` 和 `GetCount(Player player)`
  - `MaoCrest` 使用 `IsFullForPlayer(player)` 替代 `IsFull`
  - `MaxDustVar` 使用 `_owner` 获取 relic Owner 替代 `MaxDust`
- **教训**：所有 `DustManager` 无参属性（`IsFull`, `Count`, `MaxDust`, `Cards`）都只适用于本地玩家，联机代码必须使用带 `Player` 参数的版本

### 14. WishSilkEnchantment 使用 `DustManager.Cards` 导致联机分歧
- **原因**：`WishSilkEnchantment.ConvertToDustAsync()` 和 `CanEnchant()` 使用 `DustManager.Cards`（无参属性），内部调用 `GetLocalTheresaPlayer()`。Host 端检查的是 Host 自己的微尘（Ironclad 没有微尘）→ 总是认为没有副本 → 总是转化。Client 端正确检查 → 可能已有副本不转化。
- **修复**：使用 `DustManager.GetCardsForPlayer(Card.Owner)` 替代 `DustManager.Cards`
- **教训**：同上，联机代码必须使用带 `Player` 参数的 `DustManager` API

### 15. ThousandsWishPower fire-and-forget 导致联机时序分歧
- **原因**：`ThousandsWishPower` 使用 `_ = TriggerExtraLingerDelayed(...)` fire-and-forget，延迟 0.8 秒后执行额外萦绕。额外萦绕可能在 checksum 检测时一端已完成、另一端未完成（尤其是 `DustItFromAction` 中 `AutoPlay` 克隆牌的移除时机不同）。
- **修复**：
  - 移除 fire-and-forget，使用 `await TriggerExtraLinger(...)` 同步执行
  - 移除 `Cmd.Wait(0.8f)` 延迟
  - `AtTurnStartPostDraw` 增加 `await Cmd.Wait(0.3f)` 确保克隆牌动画完成
- **教训**：Hook 中的异步操作应避免 fire-and-forget，尤其是涉及卡牌状态变化的操作

### 数据流

```
抽牌 → DustDrawPatch 拦截
  ├─ 攻击/技能牌 → DustManager.AddCard() → ExhaustPile + IsDustField标记
  └─ 其他牌 → 正常加入手牌

回合开始 → MaoCrest.AfterPlayerTurnStart()
  ├─ 微尘已满 → PowerCmd.Apply<MantraPower>()
  └─ 微尘未满 → CardPileCmd.Draw() → DustDrawPatch 拦截

萦绕触发 → DustManager.DustIt()
  ├─ RNG 选牌 + 选目标
  └─ DustItFromAction() → CardCmd.AutoPlay(BlockingPlayerChoiceContext)

SPOT打出 → Spot.OnPlay()
  ├─ DamageCmd.Attack() → 触发 AfterCardPlayed
  │   └─ KazdelInFlamesEffect → ZaakathHatePower → CreatureCmd.Damage()
  └─ ChooseDustToHand() → DustManager.RemoveCard() + CardPileCmd.Add(Hand)
```

## 未复现/待观察问题

### 2026-04-24: `MemorySilkEnchantment` 第三层效果联机分歧（未复现）

**问题描述**：`MemorySilkEnchantment`（记忆丝线附魔）的第三层效果（恨意≥3时移除3层并再打出一次卡牌）可能导致联机状态分歧。

**现象**：
- Host 报告 Client 的 `ZaakathHatePower Amount = 24`
- Client 报告自己的 `ZaakathHatePower Amount = 1`
- 分歧发生在 `UNFORGIVABLE_SIN` 打出后
- 但根源可能在之前 `BORN_IN_DARKNESS` 的 `MemorySilk` 效果执行不一致

**分析**：
1. `MemorySilkEnchantment.OnPlay` 使用 `CombatState.CloneCard(Card)` 创建副本
2. 副本通过 `CardCmd.AutoPlay()` 自动打出
3. `AutoPlay` 直接调用 `OnPlayWrapper`，**不创建 `PlayCardAction`**
4. 副本打出的效果（获得恨意、移除恨意）**不通过 GameAction 同步**

**可能的分歧路径**：
```
Host 打出 BORN_IN_DARKNESS
  ├─ BornInDarkness.OnPlay: 恨意 +1
  ├─ MemorySilkEnchantment.OnPlay:
  │   ├─ 创建副本 (CloneCard)
  │   ├─ 副本 AutoPlay
  │   │   ├─ 副本 BornInDarkness.OnPlay: 恨意 +1
  │   │   └─ 副本 MemorySilkEnchantment.OnPlay: 恨意 -2 (已记忆)
  │   └─ 原牌 MemorySilk: 恨意 -3
  │
  └─ 期望最终恨意: 原值 +1 -3 +1 -2 = 原值 -3
```

如果 `MemorySilkEnchantment.OnPlay` 在某一端被跳过（如 `Card.Owner` 为 null 或 `hatePower.Amount <= 0`），则恨意移除不会执行，导致两端恨意累积差异。

**已添加的临时日志**（`MemorySilkEnchantment.cs`）：
- `OnPlay START/END` - 方法进入/退出
- `EARLY RETURN` - 提前返回的原因
- `hatePower amount` - 当前恨意层数
- `maxRemove/removeAmount` - 计算出的移除层数
- `AutoPlay completed` - 副本打出完成
- `Modifying hatePower` - 恨意移除执行及返回值

**修复尝试**：
1. 使用 `CombatState.CloneCard()` 替代 `CardModel.FromSerializable()`（已实施）
2. `AfterPlayed` 覆盖为空实现，防止 `SilkTriggerPatch` 双重触发（已实施）

**状态**：未复现，临时日志已添加。如再次遇到分歧，可通过日志定位具体原因。
